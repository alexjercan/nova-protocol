# Spike: What weapons-HUD improvements to build, and on what substrate?

- DATE: 20260708-165647
- STATUS: RECOMMENDED
- TAGS: spike, hud, weapons, torpedo, turret

## Question

Task 20260525-133022 shipped a self-sizing torpedo reticle and angular aim-assist.
What weapons-HUD improvements come next - lead/intercept pip, lock-on cue, range and
closing-speed readout, multi-target tracking, off-screen threat arrows, ammo/state,
hostile-vs-own reticles - and how should they be built so each one is not a fresh
copy of "project a world point to a UI node"? A good answer ranks the ideas by
value-over-effort using what data already exists, and picks a substrate so the set
forms one combat HUD rather than N bespoke overlays.

## Context

The HUD (`crates/nova_gameplay/src/hud`) is a set of UI-pass overlays: `health`,
`objectives`, `velocity`, and `torpedo_target`. It deliberately uses Bevy UI, not a
second `Camera2d` (a second window camera on Bevy 0.19 blacks out the 3D scene - see
the comment in `torpedo_target.rs`), and not gizmos (debug-grade, no image/text
styling). `torpedo_target` already does the core trick: `world_to_viewport` a world
anchor each frame and move an absolutely-positioned UI node there.

What data already exists to draw, for free:

- **Turret lead**: `TurretSectionAimPoint(Option<Vec3>)` is already computed by
  `lead_intercept_point` (turret_section.rs) - the intercept point for a moving
  target. Nothing draws it yet.
- **Lock + relation**: `SpaceshipPlayerTorpedoTargetEntity` is the current lock;
  committed torpedoes carry `TorpedoTargetChosen`/`TorpedoTargetEntity`. Target
  world transforms and `LinearVelocity` are queryable, so distance and closing speed
  are a subtraction away. `Health` is on targets, so a target health bar is cheap.

What does not exist yet (needs a mechanic, not just rendering): a lock *dwell* (lock
is instant today), ammo (tracked separately by task 133025), audio (task 162011), and
any faction/relation concept to colour reticles by hostile/neutral/own.

## Options considered

### What to build (ranked by value / effort, given the data above)

- **Cheap, data already exists (high feel payoff):** a turret lead/intercept pip
  (render `TurretSectionAimPoint`); a locked-target info readout (distance + closing
  speed + target health bar). Pure rendering off existing components, wasm-safe, no
  new mechanics.
- **Needs a small new mechanic:** lock-on dwell timer + acquire/lock cue (a ring that
  fills while you hold aim, plus an audio blip) - turns the instant lock into
  something satisfying and legible, but changes targeting behaviour and needs audio
  (162011); reticle differentiation by relation (hostile / your-own-torpedo /
  neutral) - needs a lightweight faction/relation notion.
- **Bigger, gameplay-forward:** off-screen target/threat direction arrows clamped to
  the screen edge; a multi-target list / subtarget cycling HUD (builds on the
  aim-assist candidate enumeration); ammo/weapon-state display (blocked on the ammo
  mechanic, 133025).

### How to build it (the substrate)

- **A. One-off systems per indicator** (what `torpedo_target` does today). Simple per
  feature, but every new indicator re-implements world-to-viewport, visibility
  lifecycle, and off-screen handling. Duplication grows with the HUD.
- **B. One generic screen-projected-indicator widget.** A reusable "world anchor ->
  UI node" layer: give it an anchor (entity or world point), it projects each frame,
  hides when the anchor is gone, and (optionally) clamps to the screen edge with a
  direction arrow when the anchor is off-screen. Lead pip, target info, and edge
  arrows all become thin consumers. More upfront work; removes the duplication and is
  a clean candidate to promote to `bevy_common_systems`.
- **C. Immediate-mode gizmos for markers.** Fast to draw, but gizmos are debug-grade
  (no sprites/text/styling) and fight the project's UI-pass HUD convention. Poor fit
  for a polished combat HUD.

## Recommendation

Build the weapons HUD as one **combat-HUD arc on a shared projection substrate**
(Option B), delivered cheap-first.

1. **Extract the substrate early (Option B).** Generalize the `torpedo_target`
   world-to-viewport trick into a reusable screen-projected-indicator widget: a
   component holding a world anchor (entity or point) plus styling, a system that
   positions/hides its UI node each frame, and optional edge-clamping with a
   direction arrow for off-screen anchors. Everything else in this arc is then a
   small consumer, and the widget is promotable to `bevy_common_systems`. Doing this
   before the third bespoke overlay is what stops the HUD from becoming N copies of
   the same projection code.

2. **Phase 1 - cheap wins on existing data (do first).** A turret lead/intercept pip
   that renders the already-computed `TurretSectionAimPoint`, and a locked-target
   info readout (distance, closing speed, target health). These are pure rendering,
   wasm-safe, high feel-per-line, and each is a first consumer of the substrate.

3. **Phase 2 - small mechanics.** Lock-on dwell + acquire/lock cue (pairs with the
   audio task 162011); reticle differentiation by relation once a minimal
   faction/relation concept exists.

4. **Phase 3 - gameplay-forward.** Off-screen target/threat edge indicators and a
   multi-target/subtarget list, both consumers of the substrate; ammo/weapon-state
   display, which belongs with the ammo mechanic (133025), not here.

Option B beats A because this arc adds at least four world-anchored indicators;
paying for one projection layer is cheaper than four. It beats C because the HUD is
deliberately UI-pass for styling and to avoid the second-camera blackout. Cheap-first
ordering means the highest feel-per-effort items (lead pip, target info) ship while
the mechanics-bearing items (dwell/cue, factions, ammo) wait on their dependencies.

## Open questions

- **Faction/relation model.** Colouring reticles hostile/neutral/own needs a
  relation concept nova does not have yet. Minimal version: hostile vs
  your-own-torpedo using existing markers. A fuller faction system is its own spike.
- **Does the lock become a dwell?** Instant lock is simplest; a dwell makes the cue
  and progress ring meaningful but changes feel. Decide when Phase 2 is planned.
- **Promote the substrate now or after a second consumer?** Extracting to
  `bevy_common_systems` is easier to judge once two nova consumers exist; keep it
  local until then (matches the promotion-catalog approach in the earlier spike).
