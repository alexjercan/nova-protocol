# Spike: How should the main menu be built and wired into the app?

- DATE: 20260711-180500
- STATUS: RECOMMENDED
- TAGS: spike, ui, menu, v0.5.0

## Question

Nova Protocol boots straight into the ship editor: `GameStates::Playing`
fires and `NovaEditorPlugin` immediately sets its private `ExampleStates::
Editor`. Task 20260711-174915 wants a real front door: a main menu with New
Game, Sandbox, Settings and Exit, laid out small-ish in the bottom-right
corner with the title "Nova Protocol", and ideally with game scenes playing
out in the background (Factorio-style). The uncertainty this spike reduces:
where does the menu live in the state machine and crate map, how do the four
entries wire into what exists today, and how is the animated background
achieved? A good answer is a task breakdown a planner can expand without
re-deriving the architecture.

## Context

Grounding facts from the code (paths as of this spike):

- `GameStates { Loading, Playing }` is defined in
  `crates/nova_gameplay/src/lib.rs:46` and is one-way; nothing hooks
  `OnExit`. `AppBuilder` (`crates/nova_core/src/lib.rs:111`) flips it to
  `Playing` when `GameAssetsStates::Loaded` is reached.
- `NovaEditorPlugin` (`crates/nova_editor/src/lib.rs:46`) reacts to
  `OnEnter(GameStates::Playing)` by entering its own private state machine
  (`Loading -> Editor -> Scenario`). The editor already owns the
  build-a-ship UI and a Play button that converts the built ship into a
  `ScenarioConfig` and fires `LoadScenario`.
- Scenario loading is fully dynamic: any system can trigger
  `LoadScenario(ScenarioConfig)` at any time
  (`crates/nova_scenario/src/loader.rs:111`), and it tears down the previous
  scenario, spawns a camera with skybox and post-processing, and fires
  `OnStartEvent`. Built-in scenarios ("asteroid_field", "asteroid_next")
  are registered into the `GameScenarios` resource during asset processing.
- UI is plain bevy_ui (buttons, flexbox nodes, observers on
  `Pointer<Press>`); no egui anywhere. The editor sidebar
  (`crates/nova_editor/src/lib.rs:363`) is the house style, and
  `DespawnOnExit(<state>)` is the established cleanup pattern.
- There is no HUD master toggle today; HUD widgets spawn per-module when
  `PlayerSpaceshipMarker` appears. The `~` HUD-level follow-up is therefore
  a separate piece of work, not part of the menu.
- Related existing tasks: ship blueprint save/load is 20260708-162014
  (v0.6.0), visual quality is 20260525-133013 (v0.6.0). Sandbox and
  Settings only need to point at these later, not implement them now.

## Options considered

### Where the menu lives

- **New `MainMenu` variant in `GameStates` + a new `nova_menu` crate
  (chosen)** - `GameStates` becomes `Loading -> MainMenu -> Playing`.
  `AppBuilder` transitions to `MainMenu` instead of `Playing` once assets
  load; menu buttons set a `GameMode` resource (NewGame | Sandbox) and flip
  to `Playing`; `NovaEditorPlugin` and scenario startup read `GameMode` on
  `OnEnter(Playing)`. Pros: the menu is app-level chrome, which is exactly
  what `GameStates` models; crate layering stays clean (menu sits beside
  `nova_editor`, above gameplay/scenario); examples that want to skip the
  menu just set `Playing` directly, so the 12 existing examples keep
  working with a one-line escape hatch (or a builder flag). Cons: touches
  the shared state enum, so `nova_core` and `nova_editor` hooks need
  auditing. Unknowns: none serious; the audit is small (only two
  `OnEnter(Playing)` hooks exist).
- **Menu as another `ExampleStates` variant inside `nova_editor`** - add
  `Menu` before `Editor`. Pros: smallest diff, one crate. Cons: wrong
  layering; the menu is not editor furniture, and New Game (scenario
  without editor) would route through the editor crate forever. Rejected.
- **Menu as a custom game plugin passed to `AppBuilder`** - keep states
  untouched, build the menu as an opt-in plugin like the examples do.
  Pros: zero risk to existing flow. Cons: the menu IS the default app now;
  hiding it behind opt-in inverts that, and the binary would diverge from
  `editor_app`. Rejected.

### UI toolkit

- **bevy_ui (chosen)** - already the house style (editor sidebar, HUD,
  status bar); flexbox handles "small panel anchored bottom-right"
  trivially (`position_type: Absolute`, `right/bottom` insets). No new
  dependency.
- **bevy_egui** - faster to iterate on settings-style forms, but a new
  dependency, a different look from the diegetic UI, and nothing else uses
  it. Rejected for now; can be revisited if the Settings screen grows real
  forms.

### Background "game scenes playing out"

- **Live ambient scenario behind the menu (chosen)** - register a
  `menu_ambience` scenario (skybox, asteroid field, a few passive AI ships
  flying by; no player, no objectives) and fire `LoadScenario` for it on
  `OnEnter(MainMenu)`. The scenario camera slow-orbits (transform-orbit
  helpers exist in bevy-common-systems). The menu panel draws over it;
  entering New Game/Sandbox just loads the next scenario, which already
  tears the old one down. Pros: real gameplay as the backdrop (the
  Factorio feel), reuses the scenario engine end to end, zero new asset
  types. Cons: simulation cost on the menu (fine, it is the same cost as
  playing); needs AI ships that do not fight the camera. Unknowns: whether
  the current AI behaves sensibly with no player present - mitigated by
  starting with drifting asteroids only and adding ships as a polish pass.
- **Static skybox + slow camera orbit only** - the degenerate version of
  the above. Kept as the explicit fallback / first milestone inside the
  same task: the menu must be usable with just skybox before any ambient
  actors are added.
- **Pre-rendered video/screenshot backdrop** - rejected: new asset
  pipeline, large files, poor fit for wasm, and it forfeits the "real
  game" charm.

### Wiring the four entries

- **New Game** - v1 loads an existing scenario ("asteroid_field") with a
  default pre-built ship (the editor already knows how to turn a
  `PlayerSpaceshipConfig` into a spawnable player; reuse that path or ship
  a canned config). The actual "fun but gentle" starter scenario is its
  own follow-up task with its own spike - the menu must not block on
  content design.
- **Sandbox** - enters the existing editor flow (`ExampleStates::Editor`),
  which already has build + Play-into-scenario. The objective-free,
  passive-enemies sandbox scenario tweak rides along in the menu task as a
  variant scenario registration ("sandbox_field": asteroid_field minus
  hostile spawns), since it is a data tweak, not a system.
- **Settings** - a placeholder sub-panel (title + Back button). Real
  content (graphics quality, keybinds, audio) is a v0.6.0 task; the panel
  exists so the button is honest.
- **Exit** - send `AppExit`. On wasm the button should be hidden (no
  process to quit); gate on `#[cfg(not(target_arch = "wasm32"))]`.

## Recommendation

Add `GameStates::MainMenu` between `Loading` and `Playing`, implement the
menu in a new `nova_menu` crate using bevy_ui (bottom-right absolute panel,
"Nova Protocol" title, four buttons), and back it with a live
`menu_ambience` scenario loaded via the existing `LoadScenario` event
(skybox-only first, ambient actors as polish). Buttons set a `GameMode`
resource and transition to `Playing`; `nova_editor` enters its editor state
only when `GameMode::Sandbox`, and New Game loads a canned scenario +
default ship. Examples keep their current behavior via a builder flag or by
setting `Playing` directly. Deferred to their own tasks: the real starter
scenario (needs its own spike), HUD visibility levels (`~` cycle), and
Settings content (v0.6.0, joins the existing visual-quality task family).

For the `~` HUD-level follow-up, the recommended shape is a
`HudVisibility { All, Minimal, None }` resource cycled by `~` (press cycles
All -> Minimal -> None -> All; no hold gesture - hold-vs-press on one key is
fiddly in bevy input and the three-state cycle is one extra press at most),
with each HUD module declaring its tier. That task is seeded
direction-level; its plan can revisit the gesture.

## Open questions

- Does the AI misbehave in a playerless scenario (targeting nothing,
  drifting off)? Resolved during the menu-background polish step; the
  skybox+asteroids fallback ships regardless.
- What exactly is the default New Game ship (which canned
  `PlayerSpaceshipConfig`)? Small decision for /plan; the editor's V1/V2
  presets are the obvious candidates.
- Whether `GameMode` should also carry "which scenario" so the future
  starter-scenario task can slot in without touching the menu again -
  planner's call, low risk either way.
