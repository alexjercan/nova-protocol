# Retro: One hit = one cue - dedup HealthApplyDamage propagation in audio + juice

- TASK: 20260709-091756
- BRANCH: fix/one-hit-one-cue (local branch by user request, squash-merged as 388b84a)
- REVIEW ROUNDS: 1 (APPROVE, no findings)

A spike-seeded bug fix that went through plan -> work -> review in one clean
round. The spike (docs/spikes/20260709-091536-combat-cue-propagation-dedup.md)
had already settled the mechanism, so the cycle's job was execution.

## What went well

- **Planning pinned the API facts before any code was written.** The plan
  phase verified `#[entity_event(propagate, auto_propagate)]` on the event,
  found `original_event_target()` on `On<E>` in the actual bevy_ecs source,
  and confirmed `SoundBank::load` works headless - so the guard and both
  tests were right on the first compile. A spike names the mechanism; the
  plan still has to verify the exact API shape.
- **Mutation-checking the new tests paid off.** Disabling the guard
  (`if false && ...`) and watching both regression tests fail with 2 cues is
  the only real proof the tests pin the bug and not just the happy path.
  Cheap (one sed + one test run) and now review-citable.
- **Prior retro lessons were applied, not just written.** From 224254:
  checked `git log master..branch` for foreign user commits before the
  squash (none this time), and forced real recompiles with `touch` instead
  of trusting sub-second check output.
- **The audio test asserts the real cue, not a proxy.** Counting `PlaySfx`
  triggers via a test observer is a stronger assertion than inspecting
  throttle stamps, and it fell out naturally from reading how
  `play_sfx_volume` works before writing the test.

## What went wrong

- **The audio test first failed with 0 sounds.** The audio observer emits via
  `Commands`, so the queued `PlaySfx` trigger only fires after a
  `world.flush()`; the juice observers mutate resources directly, which is
  why the juice test needed none. Root cause: wrote the second test by
  analogy with the first instead of by the emission path. Diagnosed in one
  run, fixed with a flush plus a comment.
- **The mutation revert silently did nothing.** `git checkout -p . </dev/null`
  answers "no" to every hunk, so the guard-disabling mutation stayed in the
  working tree; only a follow-up grep caught it before it could be committed.
  Root cause: an interactive git command in a non-interactive shell. The
  restore was redone with a targeted `sed`.

## What to improve next time

- After any scripted mutation-and-revert of the working tree, verify the
  revert with `grep`/`git diff` before moving on; never use interactive
  (`-p`) git commands from a script.
- When observer tests assert side effects, first check how the observer
  emits: direct resource writes assert immediately, `Commands`-based
  emission needs a flush (or an `app.update()`) before the assert.
