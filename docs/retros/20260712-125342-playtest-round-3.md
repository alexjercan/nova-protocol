## What went well

- The AI leash landed as a pure-function rule on an already-pure
  transition function, so the hysteresis follow-up (re-engage band) was
  a threshold helper plus one unit test instead of a system rework.
- The reviewer mutation-tested the teardown guard's regression test
  unprompted - the verify-the-fix culture from the earlier no-op-edit
  incident is now the reviewer's default, not a special request.
- Extending the placeholder-sound generator (seeded per name) meant the
  two new WAVs regenerated byte-identically alongside untouched
  existing ones - the asset pipeline lesson from the audio task held.

## What went wrong

- The objective-feedback diff celebrated scenario teardown: death/quit
  empties GameObjectives, and "every id removed" is indistinguishable
  from "everything completed" at the resource level - success chime and
  green ghosts over the objectives the player just FAILED. Root cause:
  a state RESET aliases a batch of domain EVENTS when you derive events
  by diffing state; the diff needed a semantic guard (empty = teardown)
  that the resource shape alone does not carry.
- The landing step nearly committed onto the wrong branch: the main
  checkout had been switched to v0.5.0/combat-depth by a parallel
  session, and the squash staged there before the branch check caught
  it. git reset --merge preserved the other session's uncommitted work,
  and the landing moved to a temporary master worktree.

## What to improve next time

- Deriving events from state diffs needs an explicit story for resets:
  enumerate the transitions that are NOT domain events (teardown, load,
  clear) before shipping cue/feedback logic on a diff.
- The landing rule gains a precondition: verify `git branch
  --show-current` IS the default branch before `git merge --squash`,
  and fall back to a temporary `git worktree add <tmp> master` landing
  when the main checkout is occupied by another flow.
