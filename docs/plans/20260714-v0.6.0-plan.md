# v0.6.0 plan: modding & authoring

- DATE: 2026-07-14
- BASE: master at v0.5.2 (319b26e)
- THEME: turn the code-only scenario engine into a data-driven, editable,
  showcase-able authoring platform. Ship the declarative RON scenario format
  (the "modding language"), gate its optimizations behind a benchmark, let the
  sandbox editor build & save/load scenarios, and add a screenshot-showcase
  example that generates the web site's missing media.

## Why this scope

v0.5.x made the combat feel good and shipped the web site + a 12-example testable
curriculum. The next lever is authoring: the scenario/mod engine
(`crates/nova_scenario`) is ~90% complete as *code* but is not serialized - no
`serde`, no `assets/scenarios/`, no `*.ron`. Everything content-facing (community
mods, editor save/load, a scenario builder, the flagship vertical slice) is blocked
on that one gap. So v0.6.0 has a single owning theme, "modding & authoring", split
into three reinforcing strands that all share one representation:

1. **Modding language** - the declarative RON format + its (measured) optimizations.
2. **Editor** - one consolidated task: build & save/load scenarios (a ship is just
   a scenario with a single spaceship, so ship export falls out for free).
3. **Examples/tooling** - a screenshot-showcase example that generates the ~17
   named screenshots the web site references as empty placeholders, plus a small
   SFX integration test.

Scripting (piccolo) stays backlog and gated: the prior spike parked it behind "the
declarative form provably running out of road", and we have not shipped the
declarative form yet, so we cannot have hit its ceiling. Video/GIF capture is a
backlog spike; stills cover the current web need.

Direction spike: `tasks/20260714-081636/SPIKE.md`.
Detailed design spike (type audit, asset-handle crux, optimization reality check):
`tasks/20260714-083224/SPIKE.md`.

## In scope, in execution order

IDs are tatr task IDs under `tasks/`. Priorities encode order. The format
foundation comes first because the editor builder, the built-in ports, and the
flagship scenario all consume it; the screenshot example and (mostly) the editor
run in parallel; the optimizations are measured, not assumed.

### Strand 1 - Modding language (RON format + optimizations)

1. **20260525-133029** (p80, foundation) RON format, mechanical half: add
   `serde::{Serialize, Deserialize}` across the whole config tree (full type list
   in the detailed spike; all types already derive `Clone, Debug`, a subset
   `Reflect`, none serde), then port the built-in scenarios (`asteroid_field`,
   `asteroid_next`, `menu_ambience`, `shakedown`) into `assets/scenarios/*.ron` to
   dogfood, once the loader (below) exists.
2. **20260714-083326** (p75, foundation) The design-heavy half: a `*.scenario.ron`
   `AssetLoader` + an authoring layer that resolves asset references written as
   paths/strings into live `Handle`s. This is the crux - `ScenarioConfig.cubemap`,
   `AsteroidConfig.texture` (both `Handle<Image>`) and `BeaconRenderConfig.color`
   cannot round-trip a hand-authored file. Recommended: separate authoring structs
   lowered inside the loader where a `LoadContext` exists. Gated on 133029.
3. **20260525-133028** (p70) Load `GameScenarios` from `assets/scenarios/` instead
   of code. Rides on 133029 + 083326.
4. **20260714-083331** (p45, perf/test) Scenario dispatch benchmark + profile: a
   synthetic many-handler / dense-filter scenario to get a baseline and prove where
   the time goes. Answers "are the optimizations ok"; gates the two below.
5. **20260525-133014** (p40, perf) Index handlers by event name - re-scoped, gated
   on the benchmark. Valid but small today (built-ins 1-19 handlers); insurance for
   large community mods. Dispatch loop lives in `bevy-common-systems`.
6. **20260714-083339** (p35, perf) Hot-path: intern entity-filter string keys +
   cache/compile condition eval (the higher-ROI hotspots the spike found). Gated on
   the benchmark; only the ones the numbers justify.

### Strand 2 - Editor

7. **20260714-081703** (p50, editor, spike) In-editor scenario builder: place
   objects, wire simple objectives/win-lose, save/load to/from `*.scenario.ron`.
   The SINGLE editor task (ship-blueprint save/load and a standalone UI overhaul
   were closed - ship export is an empty scenario with one ship, UI polish folds in
   here). Still a spike: explore the authoring UX before planning. The editor
   already holds the ship as `PlayerSpaceshipConfig` and lowers it to a
   `ScenarioConfig` in `test_scenario()`, so this is composition, not a rewrite -
   but only once strand 1's format + authoring layer exist.

### Strand 3 - Examples / tooling

8. **20260714-081706** (p55, example) Screenshot-showcase example: a reel of framed
   mini-scenes (built ship, orbit, combat lock, section blown off, HUD tiers, menu
   backdrop) with camera choreography, capturing the ~17 named screenshots the web
   site references (feature-*, tutorial-*, wiki-*, thumb-devlog-*). Reuses the
   existing `nova_screenshot()` / `ScreenshotPlugin` (Bevy 0.19 `Screenshot` +
   `save_to_disk`, already wired) - no new capture tech. Independent of strand 1;
   can run early. Also supports a randomized attract-mode variant.
9. **20260708-224303** (p20, test) SFX event->sound integration test - rounds out
   the testing curriculum.

## Definition of done

- A modder can write a `*.scenario.ron` file (assets referenced by path) and the
  game loads it; the built-in scenarios ship as RON, not Rust.
- The sandbox editor can save the current build to a `*.scenario.ron` and load it
  back, ship-export included.
- The screenshot-showcase example produces the web site's named screenshots.
- The scenario-dispatch benchmark exists and any optimization landed is backed by
  its numbers.
