# v0.8.0 plan: docs, tooling & content polish

- DATE: 2026-07-18
- BASE: master at v0.7.0 (unreleased at planning time; head c65f0432)
- THEME: no new game features. Pay down the debt that the fast content/feature
  releases left behind: make the docs follow the code and stop docs/ from
  collecting junk, refactor the dev tooling into a cleaner build-time-friendly
  structure (perf HTML report, Python portal/meta scripts, a per-mod audit
  report, a README you can actually start from), lengthen and polish the two
  campaigns, and fix bugs as playtest surfaces them.

## Why this scope

v0.6.0 built the authoring platform and v0.7.0 filled it with content and
polish. Both moved fast and left maintenance debt: the dev wiki drifts from the
code, docs/ root collects dated investigation notes that its own README forbids,
the README has no "how to run the tools" or getting-started section, the tooling
is a pile of Rust bins and shell/python scripts with no coherent build-time
story, the perf harness emits raw JSON/CSV that a human hand-writes into a
report, and the mod audit tool only prints a balance sheet. The two campaigns
(base Shakedown -> Broadside, and the Ledger story mod) are playable but short.

The user's stated focus for this release, verbatim in intent:

- Documentation: fix gaps, make docs follow the code, and stop docs/ collecting
  junk - the release flow should distribute stray files to their correct place.
- Bugfixes as they come up during playtest (policy, not a pre-planned list).
- More and better content: build on what exists, polish and lengthen the two
  campaigns, make them more interesting.
- Tooling and refactoring: improve the existing tools and refactor into a better
  structure. Specifically: perf as its own crate with a binary that generates a
  nice HTML report; the meta-asset generator as a Python build-time script; the
  mod portal as a Python script (the .rs is thrown away when the hosted mods API
  arrives); the mod-content tool to generate a report (audit a single mod and
  show where the errors are); and a README page documenting how to run each
  tool plus a short HOW-TO to get started.
- No new features in code for the game itself - maintenance and polishing only.

Measure-first still holds: the perf strand is reporting/ergonomics over the
existing v0.7.0 baseline, plus one flagged hitch to investigate, not new
optimization work.

State survey backing this plan: the v0.8.0 planning session's repo survey
(tooling map, docs/ state, campaign inventory, README gaps) - captured in this
doc's task set; the prior release plan is `20260716-v0.7.0-plan.md`.

## Definition of done

- The README gets you started: a HOW-TO to build/run/test and a Tools section
  documenting how to run every bin and script, with an accurate crate table.
- The dev wiki follows the code: every `web/src/wiki/dev/` page reconciled with
  current behavior; v0.7.0 features documented.
- docs/ is ephemeral: a release-time step compiles whatever accumulated in docs/
  into LESSONS.md entries and wipes the folder (20260718-175424), a guard fails
  the tag if anything but LESSONS.md remains (20260718-152225), release plans
  live in tatr tasks instead of docs/plans, and the v0.7.0 cleanup emptied the
  current junk as the interim pass.
- Perf produces a nice HTML report from a captured run, not hand-written
  markdown; nova_perf owns the report binary.
- The mod audit tool produces a per-mod report showing where each error is; the
  portal generator has a Python build-time script (Rust bin retained until
  parity is proven); the meta-gen port question is answered and applied.
- Both campaigns are meaningfully longer and more interesting, still winnable
  and losable, and the Ledger is re-published to the portal.
- Bugs found during the cycle are fixed at release priority, not queued; the
  flagged perf hitch is investigated.
