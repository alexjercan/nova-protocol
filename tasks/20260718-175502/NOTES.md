# SHIFT keybind hint + disable RCS in the mainline campaign - notes

Task: 20260718-175502. Spike: tasks/20260718-122508/SPIKE.md. RCS is a normal
granted-by-default verb (user call 2026-07-18).

## What shipped

### 1. SHIFT keybind hint (shows only when RCS is granted)

- `FlightVerbHints.rcs: VerbHint` (crates/nova_gameplay/src/input/player.rs),
  populated in `update_flight_verb_hints` like the `radar` hint: key "SHIFT"
  (fixed label, gated on the rig existing), `available = verb_granted(Rcs)`.
- Renderer (crates/nova_gameplay/src/hud/keybind_hints.rs): a 7th "RCS" row -
  `ROW_VERBS` += "RCS", `row_hint(6) -> hints.rcs`, `row(6)` in the cluster. The
  cluster is CONTEXTUAL (a row renders only while its verb is actionable), so the
  SHIFT row shows only where RCS is granted - "only when RCS enabled" for free.

### 2. RCS disabled in the mainline campaign

The base scenarios are code-GENERATED with a `content_ron_parity` test, so the
disable is authored in the BUILDERS (not the `.content.ron` directly), then the
files are regenerated (`cargo run -p nova_assets --bin content -- gen`):

- `scenario/shakedown.rs`: `DisableVerb(Rcs)` added to the `controller_gate`
  (unlike Goto/Lock/Orbit, no beat re-grants it, so it stays off all run).
- `scenario.rs` (`asteroid_field`): the player controller's `modifications`
  carries `DisableVerb(Rcs)`.
- `scenario/broadside.rs` (`player_ship()` - shared by broadside AND
  broadside_gunship): the racer's controller-cube modifications carry
  `DisableVerb(Rcs)`.

Coverage: all player-flyable base scenarios (asteroid_field, shakedown_run,
broadside, broadside_gunship). `asteroid_next` has no player ship (a bridge cut);
the menu scenarios are backdrops. RCS stays a normal verb; this is the same
per-scenario opt-out the other verbs use, easily removed at the rework.

## Difficulties / surprises

- First disabled RCS by editing the `.content.ron` directly - WRONG: those files
  are generated by the builders and guarded by `content_ron_parity`. Reverted and
  edited the builders, then regenerated. (Lesson: check whether content is
  generated before hand-editing it.)
- A heuristic script mis-identified the player controller in broadside_gunship
  (prototype names vary: `basic_controller_section` vs `racer_cube_i0_j1_k0`);
  switched to reading each builder's player controller precisely.
- The `content -- gen` + parity build was badly starved by external load
  (machine load ~48), so verification was slow - not a code issue.
